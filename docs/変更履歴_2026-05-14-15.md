## 2026-05-15（本日）の変更

---

### 1. 簡略ワークフローの実装

| | 内容 |
|--|------|
| **変更前** | 正規ワークフロー（8ステップ）のみ。所長が関与する承認ステップが多く、当現場の運用実態に合わない場合があった。 |
| **変更後** | 所長が「作業所設定」でワークフロー種別を切り替え可能。簡略ワークフロー（5ステップ）では以下が変わる: |

**正規 → 簡略の変更点**

| ステップ | 正規ワークフロー | 簡略ワークフロー |
|---------|--------------|--------------|
| ① 申請提出 | 所長へ通知 | 担当職員・所長両方に通知。**現地確認チェックを提出時に同時記入** |
| ② 所長事前承認 | 所長のみが承認 | **担当職員 または 所長どちらかが承認**（先着一方で完了） |
| ③-1 現地確認 | 協力会社が別途記入 | **提出時に記入済みのため自動スキップ** |
| ③-2 担当職員直前承認 | 担当職員が個別承認 | **①の承認で自動代替（スキップ）** |
| ④⑤ 点検・終了報告 | ④-1 協力会社サイン → ④-2 担当職員サイン → ⑤ 終了報告（3操作） | **点検時刻と終了時刻を1画面で同時入力（1操作）** |
| ⑥⑦ | 変更なし | 変更なし |

**新規追加ファイル**

| ファイル | 内容 |
|---------|------|
| `app/(protected)/dashboard/settings/page.tsx` | 作業所設定ページ（所長専用） |
| `app/(protected)/apply/[id]/simplified-pre/page.tsx` | ①事前承認画面（担当職員・所長共用） |
| `app/(protected)/apply/[id]/simplified-progress/page.tsx` | ④⑤点検・終了報告画面（協力会社） |
| `app/api/work-site-settings/route.ts` | 作業所設定 GET/PUT API |
| `app/api/applications/[id]/simplified-pre-approve/route.ts` | 事前承認 API |
| `app/api/applications/[id]/simplified-complete/route.ts` | 点検・終了報告 API |

### 2. 簡略ワークフローのメール送信バグ修正

#### バグ① — メールから事前承認するとステータスが正規フローになる

| | 内容 |
|--|------|
| **変更前** | 所長が提出メールの「事前承認する →」をクリックすると、`/api/email-approve` が `manager_pre_approved`（正規ワークフローのステータス）に変更してしまっていた。簡略ワークフローの申請が正規ワークフローの途中ステータスになり、以降の操作が不能になるバグ。 |
| **変更後** | `workflow_type === "simplified"` を判定し、簡略ワークフローの場合は `simplified_pre_approved` に遷移。`supervisor_pre_approved_at/by` も同時にセットする（③自動代替の実施）。 |
| **影響ファイル** | `app/api/applications/[id]/email-approve/route.ts` |

#### バグ② — 担当職員への提出メールに承認ボタンがない

| | 内容 |
|--|------|
| **変更前** | 簡略ワークフローでは担当職員も事前承認できるにもかかわらず、提出時の担当職員向けメールには「確認する →」リンクしかなく、承認操作への導線がなかった。 |
| **変更後** | 簡略ワークフローの場合、担当職員・所長ともに「① 事前承認する →」ボタン（`/apply/[id]/simplified-pre` へのリンク）を含むメールを送信。正規ワークフローは従来どおり所長のみメールワンクリック承認、担当職員は確認リンクのみ。 |
| **影響ファイル** | `app/api/applications/[id]/submit/route.ts` |
