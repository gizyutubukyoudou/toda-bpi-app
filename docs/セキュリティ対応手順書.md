## 1. 認証方式の概要

本システムはSupabase Authを使用したJWT（JSON Web Token）方式で認証を行う。

| トークン種別 | 有効期限 | 用途 |
|------------|---------|------|
| アクセストークン | 1時間 | APIリクエストの認証 |
| リフレッシュトークン | 60日（最終使用日から更新） | アクセストークンの自動更新 |

### セッション維持の仕組み

```
ユーザーがログイン
  └─ アクセストークン（1時間）+ リフレッシュトークン（60日）を発行
       │
       ├─ アプリ使用中: アクセストークンが期限切れ近くなると自動更新
       │    └─ リフレッシュトークンを使って新しいアクセストークンを取得
       │         └─ リフレッシュトークンも新しいものに差し替え（ローテーション）
       │
       └─ 60日間アプリを開かない場合: リフレッシュトークン期限切れ → 再ログイン必要
```

### リフレッシュトークンローテーション

リフレッシュトークンは**使用のたびに新しいものに差し替え**られ、古いものは即時無効化される。これにより、盗まれた古いリフレッシュトークンでの不正利用が検出可能となる。

## 2. 実施しているセキュリティ対策

### 2-1. 通信の保護

| 対策 | 実装状況 | 備考 |
|------|---------|------|
| HTTPS強制 | ✅ 実施済み | Vercelが自動適用 |
| HTTP→HTTPSリダイレクト | ✅ 実施済み | Vercelが自動適用 |
| HSTS（HTTP Strict Transport Security） | ✅ 実施済み | Vercelデフォルト |

### 2-2. トークン管理

| 対策 | 実装状況 | 備考 |
|------|---------|------|
| アクセストークン短命化（1時間） | ✅ 実施済み | 漏洩時の被害を最小化 |
| リフレッシュトークンローテーション | ✅ 実施済み | Supabase Auth デフォルト |
| 古いリフレッシュトークンの即時無効化 | ✅ 実施済み | 不正利用の早期検出に寄与 |
| ログアウト時のトークン全削除 | ✅ 実施済み | localStorage・サーバー両方から削除 |

### 2-3. アクセス制御

| 対策 | 実装状況 | 備考 |
|------|---------|------|
| Row Level Security（RLS） | ✅ 実施済み | DBレベルで自分のデータのみアクセス可 |
| APIルートのトークン検証 | ✅ 実施済み | 全APIエンドポイントでBearer token検証 |
| ロールベースアクセス制御 | ✅ 実施済み | contractor / supervisor / manager で画面・操作を分離 |
| 同一作業所内のデータ分離 | ✅ 実施済み | 他作業所のデータは参照不可 |

### 2-4. 運用上の制限

| 対策 | 内容 |
|------|------|
| メンバー登録の管理者制限 | 所長（manager）のみメンバー追加・削除が可能 |
| 招待制登録 | メール招待経由のみ、自己登録不可 |
| パスワードポリシー | 8文字以上（初回ログイン後に変更を推奨） |

## 3. トークン漏洩の検出方法

### 3-1. 異常の兆候

以下を検出した場合、トークン漏洩の可能性がある。

| 兆候 | 確認場所 |
|------|---------|
| 同一ユーザーが異なるIP・端末から同時アクセス | Supabase Dashboard → Auth → Users → ログイン履歴 |
| 古いリフレッシュトークンの再使用試行エラー | Supabase Dashboard → Logs → Auth logs（`invalid_refresh_token`） |
| 業務時間外の異常なアクセス | Supabase Dashboard → Logs → API logs |
| 知らない端末からのログイン成功 | Supabase Dashboard → Auth → Users |

### 3-2. 定期確認（推奨）

| 頻度 | 確認内容 | 担当 |
|------|---------|------|
| 月1回 | アクティブユーザー一覧の確認 | 所長または管理者 |
| 随時 | 不審なログインの確認（下記手順） | 所長または管理者 |

#### Supabase での確認手順

```
1. https://supabase.com にアクセス → プロジェクトを開く
2. 左メニュー「Authentication」→「Users」
3. 各ユーザーの「Last Sign In」を確認
4. 不審なユーザーは右クリック → 詳細確認
```

## 4. トークン漏洩時の対処手順

### Step 1：状況の確認（5分以内）

```
□ 何のトークンが漏洩したか確認
  - アクセストークン → 1時間で自動失効するため緊急度低
  - リフレッシュトークン → 即時対応が必要
□ 漏洩したユーザーアカウントを特定
□ 影響範囲（そのユーザーが閲覧できるデータ）を確認
```

### Step 2：該当ユーザーのセッションを即時無効化（10分以内）

**Supabase Dashboardから操作（所長・管理者）：**

```
1. Supabase Dashboard → Authentication → Users
2. 該当ユーザーを検索
3. 右端の「…」メニュー → 「Revoke all sessions」
   └─ 全デバイスからの強制ログアウト・全トークン無効化
4. 完了を確認
```

**または Malme に連絡して対応を依頼：**

```
連絡先: koushiro.tabei@malme.co.jp
```

### Step 3：パスワードリセット

```
1. Supabase Dashboard → Authentication → Users
2. 該当ユーザーを選択 → 「Send password recovery」
3. ユーザーにメールが届く → 新しいパスワードに変更
4. 変更完了を確認
```

### Step 4：被害確認・記録

```
□ 漏洩期間中に不正なアクセスや操作がなかったか確認
  - Supabase → Logs → API logs で該当ユーザーの操作履歴を確認
□ 不正な申請書の提出・承認操作があった場合は手動で取り消し
□ インシデント記録を作成（日時・対応内容・確認結果）
```

### Step 5：再発防止

```
□ 端末の紛失・盗難が原因の場合: 端末のリモートワイプを実施
□ フィッシング・マルウェアが疑われる場合: 端末のウイルス対策を確認
□ 全ユーザーへ注意喚起（必要に応じて）
```

## 5. アカウント管理ルール

### 5-1. メンバーの追加・削除

| 操作 | 実施タイミング | 担当 |
|------|------------|------|
| メンバー追加 | 新規入場時 | 所長（manager） |
| メンバー削除 | 退場・契約終了時（即日） | 所長（manager） |

> **重要**: 退場した作業員のアカウントは**当日中に削除**すること。削除しないと退場後もシステムにアクセス可能な状態が続く。

### 5-2. パスワード管理

| ルール | 内容 |
|--------|------|
| 初回ログイン | 招待メールのパスワードを使用 → 即日変更 |
| 変更頻度 | 3ヶ月に1回以上を推奨 |
| 共有禁止 | 他の作業員へのパスワード共有は禁止 |
| 端末紛失時 | 即座に所長へ報告 → セッション無効化 |

## 7. 用語説明

| 用語 | 説明 |
|------|------|
| JWT（アクセストークン） | ログイン後に発行される認証情報。短命（1時間）で、APIアクセスに使用する |
| リフレッシュトークン | アクセストークンを自動更新するための情報。長命（60日）だが使用のたびに新しいものに差し替えられる |
| RLS（Row Level Security） | データベースレベルのアクセス制御。自分のデータ以外は取得できない仕組み |
| セッション無効化 | ログイン状態を強制的に終了させること。Supabase管理画面から実施可能 |
| トークンローテーション | リフレッシュトークンを使うたびに新しいものに差し替え、古いものを無効化する仕組み |
